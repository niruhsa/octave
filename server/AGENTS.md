# Music Server ŌĆö Agent Guide

Server-side of the Music Server & App project. See the root [`AGENTS.md`](../AGENTS.md) for whole-system context. This file covers the server only.

## Purpose
Authoritative backend that hosts lossless/downloaded music and streams it to desktop/mobile clients. When online, the server is the source of truth; clients fall back to their offline cache when it is unreachable.

## Tech Stack
- **Language:** Rust (edition 2024).
- **Primary interface:** gRPC.
- **Fallback interface:** REST.
- **Crate name:** `server` (see [`Cargo.toml`](./Cargo.toml)).
- Entry point: [`src/main.rs`](./src/main.rs).

## Responsibilities
- **Streaming:** serve lossless/downloaded audio to authenticated clients.
- **Library management:** create / update / delete tracks, albums, artists, metadata.
- **Recommendations:** fingerprint-matching based music recommendation.
- **Playlists:** create / update / delete.
- **Offline support:** allow clients to download and archive content for offline use.
- **Artwork:** automatic album artwork fetching.
- **Metadata editing:** optional, manual, opt-in ŌĆö on upload and after upload completes.
- **Uploads:** accept new music from clients individually or as archives (archive file, ISO, CD, and other popular music archiving formats).
- **Ingest folder:** watch a server-side ingest dir. Files placed there (e.g. completed torrents) are **copied ŌĆö not moved** ŌĆö out and auto-organised into the library. Never delete or move the source.
- **Follow notifications:** notify users when a followed artist has a new release.
- **Audit log:** record every change with viewer access for managers/admins. Changes must be **rollback-able** (e.g. metadata edits).

## Authentication & Authorization
Both gRPC and REST use the same auth. Two mechanisms:
1. **`SECRET_KEY`** env var ŌĆö pre-shared key. Changeable at runtime via the admin UI or by an authenticated admin client.
2. **Username/password** accounts ŌĆö multiple accounts, tiered permissions.

Permission levels (each inherits the level below):
- **Admin:** modify everything, including other accounts' permissions/levels.
- **Manager:** manage the music library (create, update, delete, metadata changes). Cannot change settings or other users.
- **User:** read-only. Can still download and archive for offline use.

## Admin UI
- Optional. **Not started by default.**
- Enabled via an environment variable.
- Hosts runtime settings, including `SECRET_KEY` rotation.

## Environment Variables
- `SECRET_KEY` ŌĆö pre-shared auth key (runtime-changeable). **Required.**
- `DATABASE_URL` ŌĆö PostgreSQL connection. **Required from Phase 2 onward.**
- `GRPC_ADDR` ŌĆö gRPC bind address. Default `0.0.0.0:50051`.
- `GRPC_TLS` ŌĆö `1`/`true` to serve gRPC over TLS. Default off (plaintext). When on, `GRPC_TLS_CERT` + `GRPC_TLS_KEY` are required (a missing one is a hard config error ŌĆö TLS never silently degrades). Clients must then connect over an `https://` gRPC URL.
- `GRPC_TLS_CERT` ŌĆö PEM certificate (chain) path. Path-typed. Required when `GRPC_TLS` is on.
- `GRPC_TLS_KEY` ŌĆö PEM private key path. Path-typed. Required when `GRPC_TLS` is on.
- `REST_ADDR` ŌĆö REST bind address. Default `0.0.0.0:8080`.
- `ENABLE_ADMIN_UI` ŌĆö `1`/`true` to start the optional admin UI.
- `LIBRARY_PATH` ŌĆö organised library root. Path-typed (see below).
- `INGEST_PATH` ŌĆö copy-only ingest folder. Path-typed (see below).
- `WRITE_TAGS` ŌĆö `1`/`true` to write metadata edits back to file audio tags via `lofty`. Default off (DB stays authoritative; files untouched).
- `PRIMARY_LANGUAGE` ŌĆö display language whose spelling is shown as the canonical artist/album name when merged duplicates carry multiple spellings (e.g. English `YUQI` vs. Korean `ņÜ░ĻĖ░ ((ņŚ¼ņ×É)ņĢäņØ┤ļōż)`). Accepts a name or ISO-639 code (`English`/`en`/`eng`), normalized via `tag::normalize_language`. Default `English`. (A per-user setting later; an env var for now.)
- `FETCH_ARTWORK` ŌĆö `1`/`true` to enable automatic album-artwork fetch from the Cover Art Archive. Default off.
- `ARTWORK_PATH` ŌĆö directory where fetched cover images are cached. Path-typed. Defaults to `<LIBRARY_PATH>/.artwork` when unset.
- `FCM_ENABLED` ŌĆö `1`/`true` to enable real-time push (Firebase Cloud Messaging) of new-release notifications to followers' registered devices. Default off (clients poll instead). When on, `FCM_PROJECT_ID` + `FCM_CREDENTIALS` are both required (a missing one is a hard config error).
- `FCM_PROJECT_ID` ŌĆö Firebase project id (embedded in the FCM `messages:send` URL). Required when `FCM_ENABLED`.
- `FCM_CREDENTIALS` ŌĆö path to a Google **service-account JSON key** (used to mint the OAuth2 token for the FCM HTTP v1 API). Path-typed. Required when `FCM_ENABLED`; only the path is held, never the key bytes. **Setup walkthrough:** [`FIREBASE_NOTIFICATIONS.md`](../FIREBASE_NOTIFICATIONS.md).
- `ENV_FILE` ŌĆö explicit override for the `.env` file location.

### `.env` loading & path resolution
`Config::from_env` auto-locates a `.env` file at startup and seeds the
process environment from it. Search order:
1. `ENV_FILE` (if set).
2. Walk upward from the current working directory looking for `.env`.
3. `CARGO_MANIFEST_DIR/.env` (compile-time anchor; works for `cargo run`).
4. None ŌĆö process env is used as-is.

The directory containing the loaded `.env` becomes the **config anchor**.
All path-typed env vars (`LIBRARY_PATH`, `INGEST_PATH`, `ARTWORK_PATH`, `GRPC_TLS_CERT`, `GRPC_TLS_KEY`) resolve relative
to that anchor when not absolute, so the server behaves the same
regardless of which directory it is launched from. Copy [`.env.example`](./.env.example)
to `.env` to get started; `.env` is gitignored.

Deployment is via Docker Compose (server + PostgreSQL); see [`PLAN.md`](./PLAN.md) Phase 13.

## Conventions
- Ingest = **copy**, never move/delete source files.
- Every mutating action writes an audit log entry that supports rollback.
- gRPC is primary; keep REST as a feature-parity fallback for the same auth and operations.
- Permission checks enforced server-side on every gRPC and REST endpoint.

**Pausable uploads ŌĆö new `paused` session state (manual + auto).** An upload session can now be **paused** and **resumed**.
- **State** ([`db/models.rs`](./src/db/models.rs)): `UploadState` gains `Paused`. New `paused`/`resumed` hub event kinds.
- **Server-side stall sweeper (authoritative auto-pause).** A background task (`spawn_stall_sweeper`, started in [`main.rs`](./src/main.rs)) sweeps every **20 s** and marks any active upload idle for **Ōēź60 s** as `paused`, broadcasting a `paused` event. This is what sets `paused` when chunks simply stop arriving ŌĆö it does **not** depend on the client's own best-effort `pause` call, which fails in the very case that matters (the client's network went down, or it was killed). Idle time is `GREATEST(uploads.updated_at, MAX(upload_chunks.received_at) ?? created_at)` via `UploadRepo::pause_stale_active` (one atomic `UPDATE ŌĆ” RETURNING`), so it tracks real chunk activity yet a just-resumed session (which bumps `updated_at`) isn't immediately re-paused. The session is only created *after* the client finishes hashing (so a long local hash can't trip it), and the sweep excludes already-`paused`/terminal rows.
- **Transports**: REST `POST /uploads/:id/pause` + `/uploads/:id/resume` ([`rest/upload.rs`](./src/rest/upload.rs)); gRPC `PauseUpload` / `ResumeUpload` ([`proto/upload.proto`](./proto/upload.proto) + [`grpc/upload_svc.rs`](./src/grpc/upload_svc.rs)). Both delegate to the one service. **91 lib tests** (+1: paused is-active/parse).

**Image optimization ŌĆö fast-loading artwork (startup / upload / schedule / on-demand).** Cover + artist images are downscaled (longest side Ōēż `IMAGE_MAX_DIM`, default **800px**) and re-encoded as JPEG (`IMAGE_QUALITY`, default **82**) so the client loads them fast.
- **`ImageOptimizer`** ([`services/image_opt.rs`](./src/services/image_opt.rs)): optimized variants live in `<ARTWORK_PATH>/.optimized/<key>.jpg` ŌĆö a **parallel** cache, leaving the pristine source untouched, so re-optimizing is idempotent and never re-compresses a JPEG-of-a-JPEG. A variant is "fresh" while its mtime Ōēź the source's, so a re-upload (which rewrites the source) transparently invalidates it. Decode (jpeg/png/gif/webp via the pure-Rust `image` crate) + resize + JPEG-encode runs in `spawn_blocking`; **any failure falls back to serving the original** so a bad/corrupt image never 500s.
- **Four triggers, one path** (`ensure_optimized` / `optimize_file`): **on-demand** ŌĆö `serve_album_cover` / `serve_artist_image` optimize a not-yet-optimized image at request time, then serve + cache it; **on upload** ŌĆö the upload handlers warm the cache in the background (`warm_optimized`); **on startup + on a schedule** ŌĆö `run_optimize_pass` walks every album cover + artist image, spawned once at boot and again every `IMAGE_OPTIMIZE_INTERVAL_SECS` (default 6h; 0 disables). The pass reuses `ensure_optimized`, so repeat runs skip already-fresh variants and are cheap.
- **Low-res placeholder variant (for client blur-up):** `ensure_optimized`/`optimize_file` take a `Variant { Full, Low }`; `Low` is a tiny ~64px q58 JPEG (~1ŌĆō3 KB) cached as `<key>.lq.jpg` and served when the cover/artist-image endpoint receives `?lowres=1`. The startup/scheduled pass + the on-upload warm now generate **both** variants (`ensure_all`). The client renders the low-res instantly (blurred) while the full image loads, so artwork never pops in.
- New repo reads `AlbumRepo::all_cover_paths` / `ArtistRepo::all_image_paths`; `RestState` gains an `Option<ImageOptimizer>` (Some whenever `ARTWORK_PATH` is set). `image` crate added (default-features off ŌåÆ jpeg/png/gif/webp only, no C deps). 6 unit tests (downscaleŌåÆJPEG, no-upscale-of-small, reject-non-image, key namespacing, low-res-is-tiny, variant-suffix) ŌåÆ **90 lib tests**. (Scope: served web artwork only ŌĆö embedded audio-file art is left untouched.)

**Uploads v2 ŌĆö DB-backed, session-oriented, per-chunk-verified uploads with reports + live broadcast.**

**Duration accuracy + folder cleanup (inter-phase fix).**
- **Root cause of "wrong track length / playback drifting away from the seek bar"**: the client `<audio>` element maps a seek time to a byte offset using the *reported duration* (`time/duration ├Ś fileSize`) for formats without a container seek index (VBR MP3). When the stored duration was wrong, every seek landed on the wrong byte ŌĆö which presented as "streaming byte offset is misaligned with the timestamps," even though the byte-range serving itself (REST `GET /tracks/:id/stream` and the client `media://` local-file handler) is byte-exact (verified against `dd` and the RFC inclusive-range math). The wrong duration came from two places below.
- **Symphonia codec coverage** ([`Cargo.toml`](./Cargo.toml)): the dependency previously used Symphonia's *default* feature set (flac/ogg/vorbis/wav/pcm/mkv only). For every other format ŌĆö m4a/mp4/aac/aiff/alac ŌĆö `duration::measure_duration` returned `None`, so the cross-validation step silently fell back to lofty's header-read duration (wrong for VBR / headerless files). Now enables `aac` + `alac` + `isomp4` + `aiff` on top of the defaults.
- **MP3 duration via a dependency-free frame walker** ([`services/mp3.rs`](./src/services/mp3.rs)): Symphonia 0.6.0 pins `symphonia-bundle-mp3 = 0.6.0`, which was never published to crates.io (only `-alpha`), so the `mp3`/`mpa` features won't resolve and `measure_duration` returned `None` for MP3 ŌĆö leaving lofty's bitrate-estimate duration, which is catastrophically wrong for VBR-without-Xing (measured ~811 s for a real 132 s track in testing). `measure_mp3_duration` parses MPEG audio frame headers directly: it skips any ID3v2 tag, trusts a Xing/Info/VBRI frame-count header when present (fast path), and otherwise walks every frame summing real per-frame durations (`samples / sample_rate`) ŌĆö correct for CBR *and* VBR. `duration::measure_duration` now dispatches `.mp3`/`.mp2`/`.mp1` to this walker. Verified to agree with `ffprobe -count_frames` (5055 frames ŌåÆ 132.049 s) across CBR, VBR+Xing, and VBR-noXing encodes of the same source.
- **Lofty malformed-tag resilience** ([`services/tag.rs`](./src/services/tag.rs)): real-world MP3s with a dotted date frame (`TDRC = 2025.11.24` instead of ISO `2025-11-24`) trip a *fatal* error in lofty's ID3v2 timestamp parser even under `ParsingMode::Relaxed`, which previously made `read_tags` return `Err` and **rejected the file from ingest/scan entirely**. `read_tags` now retries with a properties-only read (`read_tags(false)`) on any read failure, so the file is still indexed (accurate duration/codec/bitrate; title/artist fall back to filename / `Unknown`). A second failure degrades to empty technical defaults rather than erroring.
- All three fixes apply on **file addition** (`ScanService::ingest_one` + `index_file`, both cross-validate the tag duration against the measured one) and on **whole-library rescan** (`refresh_durations` / `rescan_library`).
- **Rescan now refreshes technical metadata too** ([`services/scan.rs`](./src/services/scan.rs) + `TrackRepo::update_file_props` in [`db/repo.rs`](./src/db/repo.rs)/[`db/pg.rs`](./src/db/pg.rs)): the full-metadata `rescan_library` path re-writes `codec`/`bitrate_kbps`/`file_size` from the freshly-read tags in addition to title/track_no/disc_no and the Symphonia-measured duration (previously codec/bitrate/size were skipped).
- **Folder cleanup on delete** ([`services/library.rs`](./src/services/library.rs)): `delete_track` now prunes now-empty parent directories after removing the on-disk file, walking up `Album ŌåÆ Artist ŌåÆ Language` and stopping at the library root or the first non-empty directory. Because `std::fs::remove_dir` only succeeds on an empty dir, deleting an album's last track removes the album folder; deleting an artist's last album removes the artist folder; and removing a language's last artist removes the language folder ŌĆö while any folder still holding other files (cover art, other tracks) is left intact. `delete_album`/`delete_artist` cascade through `delete_track`, so deleting an album or artist removes its on-disk folder tree the same way. Boundary is enforced by canonicalising both the candidate dir and `library_root`.
- **Cover art embedded into audio file tags + auto-fetch during ingest** ([`services/tag.rs`](./src/services/tag.rs), [`services/artwork.rs`](./src/services/artwork.rs), [`services/ingest.rs`](./src/services/ingest.rs)): `tag::write_cover` embeds a cover image into any audio file via lofty's picture API (ID3v2 APIC, Ogg METADATA_BLOCK_PICTURE, etc.) using `Tag::push_picture`, replacing any existing front-cover. `ArtworkService::fetch_for_album` now calls `embed_cover_in_tracks` after caching: it resolves every track's file path and writes the fetched cover bytes into each file's tags. `IngestService` holds an optional `ArtworkService` reference (only for the remote-fetch fallback). During `organize_and_index`, if the album was just created without a `cover_path`, ingest first looks for a **cover that shipped with the upload** ŌĆö **this runs regardless of `FETCH_ARTWORK`**. `local_cover(source)` resolves (1) a **sidecar image** next to the source file ŌĆö `cover`/`folder`/`front`/`album`/`albumart`/`artwork` ├Ś `jpg`/`jpeg`/`png`/`webp`/`gif` (how loose folders and extracted archives carry art), then (2) **embedded art** in the audio file's own tags (`tag::read_embedded_cover`, prefers `CoverFront` else first picture). For archives the sidecar works because extraction fully completes before member iteration, so `cover.jpg` sits next to the tracks in the staged tree. **Key fix:** previously the whole cover step was gated behind `self.artwork.is_some()`, so with `FETCH_ARTWORK` off (the default) no cover ŌĆö not even a user-supplied `cover.jpg` ŌĆö was ever captured; and covers only landed in the `.artwork` cache, never in the library album folder. Covers thus populate for new files via the upload endpoint, the folder-watcher, and archive extracts ŌĆö user-supplied artwork wins over the remote fetch. **Single-file uploads** now carry an optional cover too: the REST `POST /upload` handler accepts a second multipart `cover` field (classified by field name `cover`/`artwork`/`art`/`image`/`thumbnail`, an `image/*` content-type, or an image extension) and the gRPC `UploadInfo` gained `cover` + `cover_filename` fields; both stage each upload in its own per-upload subdir `INGEST_PATH/.tmp/<uuid>/` (source written as `source.<ext>`, cover as `cover.<imgext>`) so `local_cover` finds the sidecar and concurrent uploads stay isolated. The Tauri client auto-detects a sidecar cover next to a picked single audio file (stems `cover`/`folder`/`front`/`album`/`albumart`/`artwork` ├Ś image exts) and threads it through `ServerClient::upload_file(cred, filename, data, cover: Option<(String, Vec<u8>)>)` (gRPC primary / REST fallback).
- **Server serves cover bytes** ([`rest/library.rs`](./src/rest/library.rs)): new `GET /albums/:id/cover` endpoint serves the cached cover file with the correct `Content-Type` (jpeg/png/gif/webp). With no `LIBRARY_PATH` configured, only absolute `file_path` values are accepted. Missing files map to `NotFound`. ExtensionŌåÆMIME table covers flac/mp3/ogg/opus/m4a/wav/alac/wv/ape; everything else streams as `application/octet-stream`.
- REST endpoint `GET /tracks/:id/stream` (+ `HEAD`) in [`rest::streaming`](./src/rest/streaming.rs):
  - `200 OK` for full-body requests; `206 Partial Content` for any satisfiable `Range:` header.
  - `416 Range Not Satisfiable` (with `Content-Range: bytes */<size>`) when the range parses but is out of bounds.
  - Malformed `Range` is ignored and the full body is served (RFC 7233 ┬¦3.1).
  - Always emits `Content-Type`, `Accept-Ranges: bytes`, `Content-Length`, and `Last-Modified`; `Content-Range` on `206`.
  - Body is built from `tokio_util::io::ReaderStream` over a `seek`+`take(len)`'d file ŌĆö no buffering of the whole file in memory.
- Pure [`rest::range::parse_range`](./src/rest/range.rs) handles `bytes=A-B`, `bytes=A-`, `bytes=-N`, whitespace, upper-bound clipping per RFC, and explicitly rejects multi-range (`bytes=0-9,20-29`) as malformed (we don't serve `multipart/byteranges`).
