# Music Server — Agent Guide

Server-side of the Music Server & App project. See the root [`AGENTS.md`](../AGENTS.md) for whole-system context. This file covers the server only.

## Purpose
Authoritative backend that hosts lossless/downloaded music and streams it to desktop/mobile clients. When online, the server is the source of truth; clients fall back to their offline cache when it is unreachable.

## Tech Stack
- **Language:** Rust (edition 2024).
- **Primary interface:** gRPC.
- **Fallback interface:** REST.
- **Crate name:** `server` (see [`Cargo.toml`](./Cargo.toml)).
- Entry point: [`src/main.rs`](./src/main.rs).

## Responsibilities
- **Streaming:** serve lossless/downloaded audio to authenticated clients.
- **Library management:** create / update / delete tracks, albums, artists, metadata.
- **Recommendations / Discover:** **behavioral** home shelves (more from artists you love, jump back in, your favorite albums, recently added — only the non-empty ones) + a seeded **radio** queue (from an artist or album), derived from play history + favorites + the library graph (no stored model). `GetHome` is per-user (`SECRET_KEY` rejected); radio is library-derived. gRPC `music.discover.v1` + REST `/discover` · `/discover/radio`. See [`services/recommendation.rs`](./src/services/recommendation.rs). _(Acoustic-fingerprint similarity — the original "fingerprint-matching" idea — is deferred to a later phase.)_
- **Playlists:** create / update / delete.
- **Offline support:** allow clients to download and archive content for offline use.
- **Artwork:** automatic album artwork fetching.
- **Storage accounting:** denormalized library-size stats — a per-entity `storage_bytes` rollup on artists/albums/podcasts plus a singleton `library_storage` breakdown (music / podcast / artwork / other). Recomputed on scan + upload and by a 24h background job ([`STORAGE_REFRESH_SECS`](#environment-variables)); served via `GetLibraryStorage` / `GET /library/storage`. Tracks also carry probed audio-quality detail (`sample_rate_hz` / `bit_depth` / `channels`) for the client's per-song media-info panel. See [`services/storage.rs`](./src/services/storage.rs).
- **Metadata editing:** optional, manual, opt-in — on upload and after upload completes.
- **Uploads:** accept new music from clients individually or as archives (archive file, ISO, CD, and other popular music archiving formats).
- **Ingest folder:** watch a server-side ingest dir. Files placed there (e.g. completed torrents) are **copied — not moved** — out and auto-organised into the library. Never delete or move the source.
- **Follow notifications:** notify users when a followed artist has a new release.
- **Play history:** record per-user "play" events (the client decides what counts) and serve recently-played + listening stats (top tracks/artists + totals over a window). Per-user; `SECRET_KEY` rejected. Private telemetry — **not** audited. The foundation for behavioral recommendations. See [`services/playhistory.rs`](./src/services/playhistory.rs).
- **Favorites:** per-user "likes" on tracks/albums/artists (polymorphic `favorites` table, nullable FKs). Add/remove (audited, mirroring follows), `is_favorite`, full-entity list reads per kind, and a `favorited_track_ids` bulk read for client heart hydration. Per-user; `SECRET_KEY` rejected. gRPC `music.favorite.v1` + REST `/{tracks,albums,artists}/:id/favorite` + `/favorites/...`. See [`services/favorites.rs`](./src/services/favorites.rs).
- **Audit log:** record every change with viewer access for managers/admins. Changes must be **rollback-able** (e.g. metadata edits).

## Authentication & Authorization
Both gRPC and REST use the same auth. Two mechanisms:
1. **`SECRET_KEY`** env var — pre-shared key. Changeable at runtime via the admin UI or by an authenticated admin client.
2. **Username/password** accounts — multiple accounts, tiered permissions.

Permission levels (each inherits the level below):
- **Admin:** modify everything, including other accounts' permissions/levels.
- **Manager:** manage the music library (create, update, delete, metadata changes). Cannot change settings or other users.
- **User:** read-only. Can still download and archive for offline use.

## Admin UI
- Optional. **Not started by default.**
- Enabled via an environment variable.
- Hosts runtime settings, including `SECRET_KEY` rotation.

## Environment Variables
- `SECRET_KEY` — pre-shared auth key (runtime-changeable). **Required.**
- `DATABASE_URL` — PostgreSQL connection. **Required from Phase 2 onward.**
- `GRPC_ADDR` — gRPC bind address. Default `0.0.0.0:50051`.
- `GRPC_TLS` — `1`/`true` to serve gRPC over TLS. Default off (plaintext). When on, `GRPC_TLS_CERT` + `GRPC_TLS_KEY` are required (a missing one is a hard config error — TLS never silently degrades). Clients must then connect over an `https://` gRPC URL.
- `GRPC_TLS_CERT` — PEM certificate (chain) path. Path-typed. Required when `GRPC_TLS` or `REST_TLS` is on.
- `GRPC_TLS_KEY` — PEM private key path. Path-typed. Required when `GRPC_TLS` or `REST_TLS` is on.
- `REST_ADDR` — REST bind address. Default `0.0.0.0:8080`. When `REST_TLS` is on, only the host is used — the port comes from `HTTPS_PORT`.
- `REST_TLS` — `1`/`true` to serve REST over TLS (HTTPS). Default off (plaintext). Reuses the gRPC `GRPC_TLS_CERT`/`GRPC_TLS_KEY` (both required — a missing one is a hard config error). When on, REST binds `HTTPS_PORT` on `REST_ADDR`'s host; clients must then use an `https://` REST URL.
- `HTTPS_PORT` — REST HTTPS port, used only when `REST_TLS` is on. Default `8443`. An invalid value is a hard config error.
- `ENABLE_ADMIN_UI` — `1`/`true` to start the optional admin UI.
- `LIBRARY_PATH` — organised library root. Path-typed (see below).
- `INGEST_PATH` — copy-only ingest folder. Path-typed (see below).
- `WRITE_TAGS` — `1`/`true` to write metadata edits back to file audio tags via `lofty`. Default off (DB stays authoritative; files untouched).
- `PRIMARY_LANGUAGE` — display language whose spelling is shown as the canonical artist/album name when merged duplicates carry multiple spellings (e.g. English `YUQI` vs. Korean `우기 ((여자)아이들)`). Accepts a name or ISO-639 code (`English`/`en`/`eng`), normalized via `tag::normalize_language`. Default `English`. (A per-user setting later; an env var for now.)
- `FETCH_ARTWORK` — `1`/`true` to enable automatic album-artwork fetch from the Cover Art Archive. Default off.
- `ARTWORK_PATH` — directory where fetched cover images are cached. Path-typed. Defaults to `<LIBRARY_PATH>/.artwork` when unset.
- `FCM_ENABLED` — `1`/`true` to enable real-time push (Firebase Cloud Messaging) of new-release notifications to followers' registered devices. Default off (clients poll instead). When on, `FCM_PROJECT_ID` + `FCM_CREDENTIALS` are both required (a missing one is a hard config error).
- `FCM_PROJECT_ID` — Firebase project id (embedded in the FCM `messages:send` URL). Required when `FCM_ENABLED`.
- `FCM_CREDENTIALS` — path to a Google **service-account JSON key** (used to mint the OAuth2 token for the FCM HTTP v1 API). Path-typed. Required when `FCM_ENABLED`; only the path is held, never the key bytes. **Setup walkthrough:** [`FIREBASE_NOTIFICATIONS.md`](../FIREBASE_NOTIFICATIONS.md).
- `PODCAST_PATH` — directory where downloaded episode audio + cached show art are stored. Path-typed (resolved against the config anchor). When unset, defaults to `<LIBRARY_PATH>/Podcasts`; if **neither** is set the podcast subsystem is disabled (endpoints 404 / gRPC `FAILED_PRECONDITION`). Added to the streaming allow-list so a downloaded episode resolves while traversal stays blocked.
- `PODCAST_REFRESH_INTERVAL_SECS` — feed refresh poller cadence in seconds (default `1800` / 30 min; `0` disables the poller, leaving on-demand `RefreshPodcast`). Floored at 60 s when the poller runs.
- `PODCAST_AUTO_DOWNLOAD_DEFAULT` — default per-show newest-N auto-download for a freshly-subscribed feed (default `0` = metadata only; audio downloaded on demand).
- `PODCASTINDEX_API_KEY` / `PODCASTINDEX_API_SECRET` — optional PodcastIndex API credentials for richer search. **Both required together** (a half-config is a hard error, like `FCM_*` / `GRPC_TLS_*`); absent → iTunes-only search (still fully functional, no key needed).
- `STORAGE_REFRESH_SECS` — cadence (seconds) of the background library-storage refresh job (default `86400` / 24 h; `0` disables the periodic pass). Each run does a *light* refresh — index new files, prune rows whose file vanished, then recompute every storage stat. A one-shot startup recompute always runs (regardless of this value) so the homepage widget has data immediately. Gated on `LIBRARY_PATH` being set. Floored at 300 s when the periodic job runs.
- `ENV_FILE` — explicit override for the `.env` file location.

### `.env` loading & path resolution
`Config::from_env` auto-locates a `.env` file at startup and seeds the
process environment from it. Search order:
1. `ENV_FILE` (if set).
2. Walk upward from the current working directory looking for `.env`.
3. `CARGO_MANIFEST_DIR/.env` (compile-time anchor; works for `cargo run`).
4. None — process env is used as-is.

The directory containing the loaded `.env` becomes the **config anchor**.
All path-typed env vars (`LIBRARY_PATH`, `INGEST_PATH`, `ARTWORK_PATH`, `GRPC_TLS_CERT`, `GRPC_TLS_KEY`) resolve relative
to that anchor when not absolute, so the server behaves the same
regardless of which directory it is launched from. Copy [`.env.example`](./.env.example)
to `.env` to get started; `.env` is gitignored.

Deployment is via Docker Compose (server + PostgreSQL); see [`PLAN.md`](./PLAN.md) Phase 13.

## Conventions
- Ingest = **copy**, never move/delete source files.
- Every mutating action writes an audit log entry that supports rollback.
- gRPC is primary; keep REST as a feature-parity fallback for the same auth and operations.
- Permission checks enforced server-side on every gRPC and REST endpoint.

**Pausable uploads — new `paused` session state (manual + auto).** An upload session can now be **paused** and **resumed**.
- **State** ([`db/models.rs`](./src/db/models.rs)): `UploadState` gains `Paused`. It is **active** (`is_active()` true) — counts toward the one-active-upload-per-user limit, is cancellable, and still **accepts + stores chunks** (so a chunk that lands during a stall isn't lost). Migration [`20260401000000_upload_paused.sql`](./migrations/20260401000000_upload_paused.sql) widens the `uploads_state_check` CHECK to include `'paused'`; `count_active_for_user` counts it.
- **Service** ([`services/uploads.rs`](./src/services/uploads.rs)): `pause(caller, id)` (active → `paused`, idempotent, owner/admin gated) + `resume(caller, id)` (`paused` → `uploading`). `put_chunk` does **not** auto-flip `paused` → `uploading` — the **client drives resume** (its first successful chunk after a stall calls `resume`), which keeps a *manual* pause consistent (the few in-flight chunks at pause time land + are stored without spuriously un-pausing). New `paused`/`resumed` hub event kinds.
- **Server-side stall sweeper (authoritative auto-pause).** A background task (`spawn_stall_sweeper`, started in [`main.rs`](./src/main.rs)) sweeps every **20 s** and marks any active upload idle for **≥60 s** as `paused`, broadcasting a `paused` event. This is what sets `paused` when chunks simply stop arriving — it does **not** depend on the client's own best-effort `pause` call, which fails in the very case that matters (the client's network went down, or it was killed). Idle time is `GREATEST(uploads.updated_at, MAX(upload_chunks.received_at) ?? created_at)` via `UploadRepo::pause_stale_active` (one atomic `UPDATE … RETURNING`), so it tracks real chunk activity yet a just-resumed session (which bumps `updated_at`) isn't immediately re-paused. The session is only created *after* the client finishes hashing (so a long local hash can't trip it), and the sweep excludes already-`paused`/terminal rows.

**Image optimization — fast-loading artwork (startup / upload / schedule / on-demand).** Cover + artist images are downscaled (longest side ≤ `IMAGE_MAX_DIM`, default **800px**) and re-encoded as JPEG (`IMAGE_QUALITY`, default **82**) so the client loads them fast.
- **`ImageOptimizer`** ([`services/image_opt.rs`](./src/services/image_opt.rs)): optimized variants live in `<ARTWORK_PATH>/.optimized/<key>.jpg` — a **parallel** cache, leaving the pristine source untouched, so re-optimizing is idempotent and never re-compresses a JPEG-of-a-JPEG. A variant is "fresh" while its mtime ≥ the source's, so a re-upload (which rewrites the source) transparently invalidates it. Decode (jpeg/png/gif/webp via the pure-Rust `image` crate) + resize + JPEG-encode runs in `spawn_blocking`; **any failure falls back to serving the original** so a bad/corrupt image never 500s.
- **Four triggers, one path** (`ensure_optimized` / `optimize_file`): **on-demand** — `serve_album_cover` / `serve_artist_image` optimize a not-yet-optimized image at request time, then serve + cache it; **on upload** — the upload handlers warm the cache in the background (`warm_optimized`); **on startup + on a schedule** — `run_optimize_pass` walks every album cover + artist image, spawned once at boot and again every `IMAGE_OPTIMIZE_INTERVAL_SECS` (default 6h; 0 disables). The pass reuses `ensure_optimized`, so repeat runs skip already-fresh variants and are cheap.
- **Low-res placeholder variant (for client blur-up):** `ensure_optimized`/`optimize_file` take a `Variant { Full, Low }`; `Low` is a tiny ~64px q58 JPEG (~1–3 KB) cached as `<key>.lq.jpg` and served when the cover/artist-image endpoint receives `?lowres=1`. The startup/scheduled pass + the on-upload warm now generate **both** variants (`ensure_all`). The client renders the low-res instantly (blurred) while the full image loads, so artwork never pops in.

**Duration accuracy + folder cleanup (inter-phase fix).**
- **Root cause of "wrong track length / playback drifting away from the seek bar"**: the client `<audio>` element maps a seek time to a byte offset using the *reported duration* (`time/duration × fileSize`) for formats without a container seek index (VBR MP3). When the stored duration was wrong, every seek landed on the wrong byte — which presented as "streaming byte offset is misaligned with the timestamps," even though the byte-range serving itself (REST `GET /tracks/:id/stream` and the client `media://` local-file handler) is byte-exact (verified against `dd` and the RFC inclusive-range math). The wrong duration came from two places below.
- **Symphonia codec coverage** ([`Cargo.toml`](./Cargo.toml)): the dependency previously used Symphonia's *default* feature set (flac/ogg/vorbis/wav/pcm/mkv only). For every other format — m4a/mp4/aac/aiff/alac — `duration::measure_duration` returned `None`, so the cross-validation step silently fell back to lofty's header-read duration (wrong for VBR / headerless files). Now enables `aac` + `alac` + `isomp4` + `aiff` on top of the defaults.
- **MP3 duration via a dependency-free frame walker** ([`services/mp3.rs`](./src/services/mp3.rs)): Symphonia 0.6.0 pins `symphonia-bundle-mp3 = 0.6.0`, which was never published to crates.io (only `-alpha`), so the `mp3`/`mpa` features won't resolve and `measure_duration` returned `None` for MP3 — leaving lofty's bitrate-estimate duration, which is catastrophically wrong for VBR-without-Xing (measured ~811 s for a real 132 s track in testing). `measure_mp3_duration` parses MPEG audio frame headers directly: it skips any ID3v2 tag, trusts a Xing/Info/VBRI frame-count header when present (fast path), and otherwise walks every frame summing real per-frame durations (`samples / sample_rate`) — correct for CBR *and* VBR. `duration::measure_duration` now dispatches `.mp3`/`.mp2`/`.mp1` to this walker. Verified to agree with `ffprobe -count_frames` (5055 frames → 132.049 s) across CBR, VBR+Xing, and VBR-noXing encodes of the same source.
- **Lofty malformed-tag resilience** ([`services/tag.rs`](./src/services/tag.rs)): real-world MP3s with a dotted date frame (`TDRC = 2025.11.24` instead of ISO `2025-11-24`) trip a *fatal* error in lofty's ID3v2 timestamp parser even under `ParsingMode::Relaxed`, which previously made `read_tags` return `Err` and **rejected the file from ingest/scan entirely**. `read_tags` now retries with a properties-only read (`read_tags(false)`) on any read failure, so the file is still indexed (accurate duration/codec/bitrate; title/artist fall back to filename / `Unknown`). A second failure degrades to empty technical defaults rather than erroring.
- All three fixes apply on **file addition** (`ScanService::ingest_one` + `index_file`, both cross-validate the tag duration against the measured one) and on **whole-library rescan** (`refresh_durations` / `rescan_library`).
- **Rescan now refreshes technical metadata too** ([`services/scan.rs`](./src/services/scan.rs) + `TrackRepo::update_file_props` in [`db/repo.rs`](./src/db/repo.rs)/[`db/pg.rs`](./src/db/pg.rs)): the full-metadata `rescan_library` path re-writes `codec`/`bitrate_kbps`/`file_size` from the freshly-read tags in addition to title/track_no/disc_no and the Symphonia-measured duration (previously codec/bitrate/size were skipped).
- **Folder cleanup on delete** ([`services/library.rs`](./src/services/library.rs)): `delete_track` now prunes now-empty parent directories after removing the on-disk file, walking up `Album → Artist → Language` and stopping at the library root or the first non-empty directory. Because `std::fs::remove_dir` only succeeds on an empty dir, deleting an album's last track removes the album folder; deleting an artist's last album removes the artist folder; and removing a language's last artist removes the language folder — while any folder still holding other files (cover art, other tracks) is left intact. `delete_album`/`delete_artist` cascade through `delete_track`, so deleting an album or artist removes its on-disk folder tree the same way. Boundary is enforced by canonicalising both the candidate dir and `library_root`.
- **Cover art embedded into audio file tags + auto-fetch during ingest** ([`services/tag.rs`](./src/services/tag.rs), [`services/artwork.rs`](./src/services/artwork.rs), [`services/ingest.rs`](./src/services/ingest.rs)): `tag::write_cover` embeds a cover image into any audio file via lofty's picture API (ID3v2 APIC, Ogg METADATA_BLOCK_PICTURE, etc.) using `Tag::push_picture`, replacing any existing front-cover. `ArtworkService::fetch_for_album` now calls `embed_cover_in_tracks` after caching: it resolves every track's file path and writes the fetched cover bytes into each file's tags. `IngestService` holds an optional `ArtworkService` reference (only for the remote-fetch fallback). During `organize_and_index`, if the album was just created without a `cover_path`, ingest first looks for a **cover that shipped with the upload** — **this runs regardless of `FETCH_ARTWORK`**. `local_cover(source)` resolves (1) a **sidecar image** next to the source file — `cover`/`folder`/`front`/`album`/`albumart`/`artwork` × `jpg`/`jpeg`/`png`/`webp`/`gif` (how loose folders and extracted archives carry art), then (2) **embedded art** in the audio file's own tags (`tag::read_embedded_cover`, prefers `CoverFront` else first picture). A found local cover is written **into the library album folder** as `cover.<ext>` (next to the organised tracks, via `write_cover_to_library`), the album row's `cover_path` is pointed at that on-disk file (audited `update_album`), and the art is embedded into every track in the album. Only when **no** local cover exists *and* `FETCH_ARTWORK` is on does a background `tokio::spawn` fall back to the remote `ArtworkService::fetch_for_album` (MusicBrainz + CAA, which caches under `ARTWORK_PATH`). For archives the sidecar works because extraction fully completes before member iteration, so `cover.jpg` sits next to the tracks in the staged tree. **Key fix:** previously the whole cover step was gated behind `self.artwork.is_some()`, so with `FETCH_ARTWORK` off (the default) no cover — not even a user-supplied `cover.jpg` — was ever captured; and covers only landed in the `.artwork` cache, never in the library album folder. The Tauri client auto-detects a sidecar cover next to a picked single audio file (stems `cover`/`folder`/`front`/`album`/`albumart`/`artwork` × image exts) and threads it through `ServerClient::upload_file(cred, filename, data, cover: Option<(String, Vec<u8>)>)` (gRPC primary / REST fallback).

**Phase 4 — Streaming**
- New [`services::StreamingService`](./src/services/streaming.rs): resolves a track id to a safe on-disk path, gated by `Identity` (any authed user). Path resolution policy: `Track.file_path` is treated as absolute or relative-to-`LIBRARY_PATH`; `canonicalize` is applied and the result **must** live under the canonical library root or the request is denied (`PermissionDenied`). With no `LIBRARY_PATH` configured, only absolute `file_path` values are accepted. Missing files map to `NotFound`. Extension→MIME table covers flac/mp3/ogg/opus/m4a/wav/alac/wv/ape; everything else streams as `application/octet-stream`.
- REST endpoint `GET /tracks/:id/stream` (+ `HEAD`) in [`rest::streaming`](./src/rest/streaming.rs):
  - `200 OK` for full-body requests; `206 Partial Content` for any satisfiable `Range:` header.
  - `416 Range Not Satisfiable` (with `Content-Range: bytes */<size>`) when the range parses but is out of bounds.
  - Malformed `Range` is ignored and the full body is served (RFC 7233 §3.1).
  - Always emits `Content-Type`, `Accept-Ranges: bytes`, `Content-Length`, and `Last-Modified`; `Content-Range` on `206`.
  - Body is built from `tokio_util::io::ReaderStream` over a `seek`+`take(len)`'d file — no buffering of the whole file in memory.
- Pure [`rest::range::parse_range`](./src/rest/range.rs) handles `bytes=A-B`, `bytes=A-`, `bytes=-N`, whitespace, upper-bound clipping per RFC, and explicitly rejects multi-range (`bytes=0-9,20-29`) as malformed (we don't serve `multipart/byteranges`).
- Transcoder stub trait (PLAN §4) added but unimplemented.
- Tests: 7 range-parser unit tests + 5 streaming-service tests (relative-without-root rejection, relative-inside-root resolution, dotdot traversal blocked, missing file → NotFound, MIME mapping). All 21 lib tests pass.
- Verified end-to-end against the running server: `200`/`206`/`416`/`401` all correct, byte content matches expected pattern at multiple offsets.
- **Fixed a latent routing bug** discovered during Phase 4 smoke testing: every path-parameter route in `rest/library.rs` (and the new streaming route) used axum 0.8 `{id}` syntax, but the crate is pinned to axum 0.7 / matchit 0.7 which expects `:id`. Migrated all path params from `{id}` → `:id`. `/tracks/:id`, `/albums/:id/...`, `/artists/:id/...` etc. now actually match.
