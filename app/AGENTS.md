# Music App (Client) — Agent Guide

Client-side of the Music Server & App project. See the root [`AGENTS.md`](../AGENTS.md) for whole-system context and [`server/AGENTS.md`](../server/AGENTS.md) for the backend. This file covers the client app only.

## Purpose
Cross-platform music player that streams from the remote server when online and falls back to a local offline cache when the server is unreachable. Functionality comparable to [Poweramp](https://powerampapp.com/) / Spotify, with the server acting as authority when available.

## Tech Stack
- **Framework:** [Tauri 2](https://tauri.app/) ([Rust docs](https://docs.rs/tauri/2.11.3/tauri/)).
- **Frontend:** React 19 + TypeScript, bundled with Vite 8.
- **Rust core (`src-tauri/`):** native shell, plugins, and `#[tauri::command]` handlers bridging frontend ↔ native.
- **Package name:** `octave` (npm) / `octave_lib` (Rust lib).
- **App identifier:** `dev.niruhsa.octave`.

## Target Platforms
- **Minimum:** desktop (Windows/macOS/Linux) and Android — must cross-compile to both.
- **Best effort:** iOS.

## Layout
- `src/` — React/TS frontend. Entry: [`src/main.tsx`](./src/main.tsx) → [`src/App.tsx`](./src/App.tsx).
- `src-tauri/` — Rust native side. Entry: [`src-tauri/src/lib.rs`](./src-tauri/src/lib.rs) (`run()`), [`src-tauri/src/main.rs`](./src-tauri/src/main.rs).
- `src-tauri/tauri.conf.json` — app config (window, bundle, dev/build commands, identifier).
- `src-tauri/capabilities/default.json` — permission/capability grants.
- `public/` — static assets served as-is.
- `index.html` — Vite HTML entry.

## Responsibilities (client side)
- **Streaming playback** from the server with gapless/queue UX.
- **Offline cache (SQLite):** a *partial local copy* of the server DB holding **only downloaded items** — track/album/artist metadata, playlist references, and album-art cover paths for downloaded content. Never the full catalog. Play from cache when the server is offline; server (PostgreSQL) is authority when online.
- **Auth:** log in via `SECRET_KEY` or username/password; store credentials securely; respect permission tier (Admin / Manager / User) returned by server.
- **Library browse/search:** artists, albums, tracks, playlists.
- **Playlists:** create / update / delete (synced through server when online).
- **Uploads:** send new music to the server — individually or as archives (archive file, ISO, CD, other popular formats).
- **Metadata editing:** optional, manual, opt-in UI for managers/admins.
- **Follow + notifications:** follow artists; surface new-release notifications.
- **Admin/manager UI:** expose settings & user management only when the logged-in account has permission.
- **Audit/rollback views:** managers/admins can view change logs and trigger rollbacks (server-enforced).

## Server Communication
- Backend is **gRPC primary, REST fallback**. Mirror that preference in the client transport layer.
- All requests carry auth (`SECRET_KEY` or session from username/password).
- Treat server as source of truth when reachable; degrade gracefully to offline cache otherwise.
- Heavy/native work (filesystem cache, secure credential storage, downloads) belongs in the Rust side via `#[tauri::command]`; keep the React layer for UI/state.

## Conventions
- Frontend in TypeScript with `strict` mode (see [`tsconfig.json`](./tsconfig.json)) — no unused locals/params, no fallthrough cases.
- Expose native capabilities through explicit Tauri commands + capability grants in `src-tauri/capabilities/`; do not widen permissions beyond need.
- Permission-gate every admin/manager UI affordance; never assume client-side checks are sufficient — server enforces.
- Keep dev server on fixed port `1420` (Tauri requirement; see `vite.config.ts`).

**Crash-safe resumable uploads — copy-to-staging so a force-quit can't lose source access.** If the OS evicts/force-quits the app mid-upload, the upload **resumes automatically** on next launch from where it left off — and it no longer falsely reports "files changed" on Android.
- **Root cause (fixed)**: picked Android sources are SAF `content://` URIs whose read grant is **revoked on process death** (a force-quit drops even a persisted grant in practice). The earlier `takePersistableUriPermission` approach couldn't guarantee re-read, so resume read 0 bytes / errored → the hash mismatched → instant false "corrupted".
- **Fix — stage to app-private storage** ([`commands/upload_commands.rs`](./src-tauri/src/commands/upload_commands.rs)): at upload start each `content://` source is **copied** into `<app_data_dir>/upload_staging/<uuid>/<file_index>.<ext>` (always accessible; survives a kill) while the SAF grant is still valid. The upload then **reads chunks from the staged file** (`read_range`, a seek+read on a blocking thread) — so whole files never sit in RAM (also fixes the long-noted "buffer the whole file" memory issue). Desktop paths persist, so they're used directly (no copy).
- **Local persistence**: a **resume manifest** (`upload_resume.json`, atomic temp+rename) records `{ upload_id, job_id, staging_key, files: [{ path, name_hint, hash }] }` where `path` is the **staged copy** (or the desktop original). The server owns the chunk map + received-set (fetched via `get_upload` on resume). On every terminal outcome `clear_resume_state` removes the manifest **and** the staging dir; orphaned staging (crash before cleanup) is swept on next startup (`sweep_staging`, keeping the resumable one).
- **Resume on startup**: `uploads_resume_pending` (from [`App.tsx`](./src/App.tsx) when a session becomes available; `resuming`-flag guarded against double-spawn) runs `run_resume_job`: (1) re-reads + re-hashes each staged/source file vs the manifest, (2) on **any** mismatch/unreadable file **cancels the session as corrupted** with a clear reason, (3) else fetches the view and (4) hands to the shared **`drive_upload`** that uploads only the **missing** chunks (reading them from the staged files) — no re-upload, no re-hash-then-restart. `run_job` (fresh upload) routes through the same `drive_upload`. A new `ChunkSend::ReadError` fails fast if a staged file vanishes mid-upload. (`persist_uri_access` is kept as belt-and-suspenders but staging is what makes resume reliable.)
- **Resync a server-side pause on resume**: if the server's stall sweeper marked the session `paused` while the client was force-quit, `drive_upload` now **re-asserts `uploading`** when it picks the session up (a fresh resume `JobCtl` has `auto_paused = false`, so the Ok-arm's client-driven resume wouldn't fire on its own). It calls `resume_upload` once on entry when `view.state == "paused"` (best-effort; on an offline-at-start race it arms `auto_paused` so the first successful chunk re-asserts it) — fixing "the server stays Paused after a resume until you manually pause/unpause". Server-side `put_chunk` still does **not** auto-resume (keeps a *manual* pause consistent).
- **Errors**: a transient/offline `get_upload` keeps the manifest+staging (retries next launch); a gone/forbidden session clears them. Verify shows "Verifying file N/M"; the cancel reason surfaces on the [`Upload`](./src/routes/Upload.tsx) card. `cargo test` (**44** lib; manifest test covers `staging_key` + pre-staging back-compat) + clippy clean; `cargo check` host **and** `--target aarch64-linux-android` clean; `tsc` + `npm run build` clean.

**Pausable uploads — manual pause/resume + auto-pause on a ≥1 min stall.** An in-flight upload can be paused (and resumed) by the user, and auto-pauses when chunk uploads stall/fail for ≥1 minute — resuming automatically when a chunk lands again. This replaces the old "give up + cancel after 3 retries" behaviour (a network blip now **pauses** instead of failing).
- **Resilient chunk send** ([`commands/upload_commands.rs`](./src-tauri/src/commands/upload_commands.rs)): `send_chunk_resilient` retries transient failures **indefinitely** with capped exponential backoff (500 ms → 10 s), parks while manually paused (250 ms poll), stamps `last_success_ms` on success, and returns `Cancelled`/`AlreadyComplete` on a terminal session (polled every 4 failures) instead of erroring. A per-job `JobCtl` (managed `UploadControl` slot — one upload at a time) holds the pause/cancel atomics + last-success clock, shared between the chunk futures, a **stall monitor** task (every 5 s; on ≥`STALL_THRESHOLD_MS`=60 s with no success it sets `auto_paused`, best-effort `pause_upload`s the server, and updates the UI badge + foreground notification), and the commands. **Auto-resume** is client-driven: the driving loop's success arm clears `auto_paused` and calls `resume_upload` on the first chunk after a stall. A `JobCtlGuard` clears the slot + stops the monitor on every exit path.
- **Commands**: `uploads_pause` / `uploads_resume` (flip the local job flag for an immediate stop/restart, emit a `paused`/`resumed` progress event + notification, then call the server); `uploads_cancel` now also flips the local cancel flag so the job stops promptly. `ProgressEvent` gains `paused` / `pause_reason` ("manual" | "stalled").
- **Transport**: `pause_upload` / `resume_upload` across [`transport/{grpc,rest,client}.rs`](./src-tauri/src/transport/) (gRPC-primary / REST-fallback) + [`auth/manager.rs`](./src-tauri/src/auth/manager.rs); the proto regenerates `PauseUpload`/`ResumeUpload` from the server's `upload.proto`.
- **UI**: the upload store ([`uploads/useUploads.ts`](./src/uploads/useUploads.ts)) overlays `paused`/`pauseReason` onto live progress without clobbering the byte counters; [`Upload.tsx`](./src/routes/Upload.tsx) shows a **⏸ Paused** badge (+ "· stalled") and a **Pause / Resume** button next to Cancel; the [`/uploads`](./src/routes/Uploads.tsx) reports list + detail gain a `paused` filter chip, a sky state pill, and per-row / header Pause·Resume actions. `UploadLifecycle` + `UploadLiveEvent.kind` gain `paused`/`resumed`.

**Upload preparing-phase shows per-file, per-stage progress (no more silent "1/10" stall).** The pre-upload prepare loop ([`commands/upload_commands.rs`](./src-tauri/src/commands/upload_commands.rs) `run_job`) read + hashed every file while emitting **no** progress, so the UI froze on "Preparing 1/10 files" then jumped straight to "Uploading…" (a multi-file album spends real time hashing). It now emits a status per file **per stage** — **"Reading file N/M"** then **"Hashing file N/M"** — to both the in-app progress card and the OS / foreground notification (via a new `prepare_status` helper), so the counter visibly advances 1→2→…→M. `prepare_file` was reworked to a **single pass** over the bytes (folding the whole-file SHA-256 in incrementally instead of a second full pass) that invokes an `on_progress(done, total)` callback per chunk; the loop streams **throttled (150 ms) intra-file percent** ("Hashing file 3/10 · 45%") to the UI (frontend-only — too frequent for the notification) so even a single large file shows movement instead of stalling. A final **"Starting upload…"** bridges the session-declare round-trip. Frontend ([`Upload.tsx`](./src/routes/Upload.tsx)) now renders `progress.message` during the scanning phase (the `message` field already existed on `UploadProgressEvent`) instead of a static "Preparing…". `cargo test` (41 lib, `prepare_file` test updated to assert the progress ticks) + clippy clean; `cargo check` host **and** `--target aarch64-linux-android` clean; `tsc --noEmit` clean.

**Background uploads survive backgrounding / screen-lock (Android foreground service).** Uploads no longer **instantly fail** when the app is backgrounded or the phone is locked (screen off or display-on) — the previously-deferred "true Android foreground service for uploads" (Phase 8 note) is now implemented.
- **Root cause:** the upload job ([`commands/upload_commands.rs`](./src-tauri/src/commands/upload_commands.rs) `run_job`) is already fully native (a tokio task driving the chunked gRPC/REST transfer — the WebView is not involved), but it ran with **no foreground service**. The existing `.ongoing()` progress notification is *not* a foreground service — it only blocks swipe-dismiss; it grants the process **no** background execution. So the moment the app lost foreground, Android's background-network restriction + Doze severed the in-flight request, the 3 chunk retries exhausted in ~1 s, and `run_job` cancelled the session → "instantly fails".
- **Fix — `dataSync` foreground service** ([`UploadService.kt`](./src-tauri/gen/android/app/src/main/java/dev/niruhsa/octave/UploadService.kt) + [`UploadServicePlugin.kt`](./src-tauri/gen/android/app/src/main/java/dev/niruhsa/octave/UploadServicePlugin.kt)), modelled on the existing playback [`MediaService`](./src-tauri/gen/android/app/src/main/java/dev/niruhsa/octave/MediaService.kt). While an upload runs it: (1) posts a **persistent, non-dismissible** `dataSync` foreground notification (ongoing, no delete intent, a determinate/indeterminate progress bar) removed only when the upload finishes; (2) holds a **partial wake lock** so the CPU keeps running with the screen off; (3) holds a **WiFi lock** (`WIFI_MODE_FULL_LOW_LATENCY`) so the radio doesn't power down on lock — the specific cause of dropped transfers when the screen turns off. Foreground importance also exempts the process from background-network throttling + Doze.
- **Bridge** ([`upload_session.rs`](./src-tauri/src/upload_session.rs), mirroring [`media_session.rs`](./src-tauri/src/media_session.rs)): an app-inline Tauri plugin binding the Kotlin `UploadServicePlugin` via `register_android_plugin`; `start` / `update` / `stop` go Rust→Kotlin through `run_mobile_plugin` (no JS/ACL involved, same as the media bridge). `run_job` calls `start` **first thing** (while still foreground — Android 12+ forbids starting an FGS from the background) and installs an **RAII `UploadForegroundGuard`** that calls `stop` on **every** exit path (early return, error/cancel, normal completion, panic-unwind), so the service + locks + notification can't leak. Per-tick progress is pushed via `update` (throttled ~400 ms). On Android the in-progress notification is now owned by the FGS (the desktop `tauri-plugin-notification` ongoing notification path is unchanged); the one-shot completion notification is unchanged. All calls **no-op on desktop / iOS** (handle never bound).
- **Manifest** ([`AndroidManifest.xml`](./src-tauri/gen/android/app/src/main/AndroidManifest.xml)): `FOREGROUND_SERVICE_DATA_SYNC` + `WAKE_LOCK` permissions and the `UploadService` (`foregroundServiceType="dataSync"`). No new Gradle deps (NotificationCompat / PowerManager / WifiManager are already on the classpath). `POST_NOTIFICATIONS` (already present) gates only notification *visibility* — the service keeps the upload alive even if denied.
- Verified: `cargo check` host **and** `--target aarch64-linux-android` clean (both cfg branches), `cargo test` (41 lib) green, Gradle `compileArm64DebugKotlin` clean, and the merged manifest carries the new permissions + service. **On-device caveats (need a full `tauri android dev/build`, not a JS reload):** end-to-end background/screen-off upload survival, and that on **Android 14+** the user *can* swipe an FGS notification away (OS-level change — the service keeps running regardless; not app-overridable) and `dataSync` carries a ~6 h/day cumulative cap (ample for music uploads).

**Uploads: parallel chunk upload (`CHUNK_CONCURRENCY = 4`).** The upload job ([`commands/upload_commands.rs`](./src-tauri/src/commands/upload_commands.rs)) now sends up to **4 chunks of a file concurrently** instead of one-at-a-time — `futures_util::stream::iter(missing).buffer_unordered(4)` — a big win on high-latency links where each 128 KiB chunk previously serialized a full round-trip. **No server change:** the server already accepts concurrent chunks safely (per-chunk hash verify, atomic `received_chunks` recompute inside the `UPDATE`, and single-finalizer election via its `finalizing` guard, so two "last" chunks can't double-ingest).
- Each chunk future **owns its inputs** so the spawned job stays `Send + 'static` (borrowing locals tripped a `buffer_unordered` HRTB error): the file bytes are shared via a cheap `Arc<Vec<u8>>` clone and a chunk-sized copy is sliced off *inside* the future, so at most `CHUNK_CONCURRENCY × CHUNK_SIZE` (~512 KiB) chunk buffers are live at once — no memory regression vs the sequential path. The credential + `upload_id` are cloned per future (cheap).
- Acks are **consumed sequentially** in the driving loop, so the progress counters stay single-writer (no locking); resume (skip already-received chunks) + the cancel-on-unrecoverable-error path are unchanged.
- Once chunks race, a chunk can fail *after* the session finalized (a lost ack + a retry hitting the now-`Completed` session); a new `session_completed()` re-checks the upload state on a chunk error so that benign case is treated as success, not a spurious failure. `cargo test` (41 lib) + clippy clean.

**Phase 8 — Uploads: chunked & resumable + background jobs + notifications (update).** Uploads are **chunked and resumable**, run as **background jobs**, and integrate OS notifications.
- **Chunked protocol (server + client).** New server REST endpoints in [`server/src/rest/upload.rs`](../server/src/rest/upload.rs): `POST /upload/init` (declares `{ filename, hash, total_size, chunk_size, total_chunks, chunks:[{index,start,end}] }`; keyed by the file **hash** so any device resumes the same session; returns the id + chunks already present), `POST /upload/chunk/:id/:index` (raw chunk bytes; the chunk that completes the set triggers **reassembly** — concatenate in order, **verify the SHA-256 against the declared hash**, then run the existing ingest: archive → multi-file, else single track), and `GET /upload/status/:id` (received/missing chunks + the ingest result once done). Chunks live under `<ingest_root>/.uploads/<hash>/chunks/<index>.part`; presence on disk is the source of truth for "received", so resume survives a server restart. A `create_new(".assembling")` lock elects a single assembler; on success the chunk bytes are dropped and a `result.json` kept for status polling.
- **Client** ([`commands/upload_commands.rs`](./src-tauri/src/commands/upload_commands.rs) `upload_chunked` + transport `upload_init`/`upload_chunk`/`upload_status`): hashes the file (SHA-256), `init`s to learn which chunks the server already has, uploads only the **missing** chunks (4 MiB each), and emits chunk-granular `upload-progress` so the UI shows a **real per-file byte bar**. `upload_files(items)`/`upload_folder(dir)` still spawn a background job (returns a `jobId`) and drive notifications; on completion `upload-complete` fires. (The old single-shot gRPC/REST `upload_file` chain is left in place but unused.)
- **Notifications** use two Android channels: progress is a single notification (fixed id) on a **Low-importance, `vibration=false`** channel + `.silent()` + `.ongoing()` — re-`show()`n per chunk (throttled ~400 ms) it updates **in place without re-alerting**; on completion the progress notification is cleared (`remove_active`) and a **new** id is posted on a **Default-importance** channel so it alerts **exactly once**. (Desktop reuses the id / no channels.) The plugin has **no native progress bar**, so the body shows a 10-cell **text bar + overall %** (file-weighted: a single file shows that file's progress, a batch shows total across all files) **+ average upload speed**. The same job-wide average speed (`bytesPerSec` in the progress event) is shown on the [`/upload`](./src/routes/Upload.tsx) page during the upload.
- **File reading is native (Rust), never the WebView.** The [`/upload`](./src/routes/Upload.tsx) route passes picked paths/URIs straight to `upload_files`; the job reads each via the **`tauri-plugin-fs` Rust API** (`app.fs().read(FilePath)`, blocking thread) — a normal path on desktop, a SAF `content://` URI via the Android ContentResolver (fixes the earlier crash where JS `readFile`→`writeFile` blew the WebView heap). Content-URI names are unreliable (tail = opaque doc id like `msf%3A13974`), so `name_hint`/`determine_filename` percent-decode + sniff the format from magic bytes (FLAC/OGG/WAV/M4A/AIFF/MP3/ZIP/gzip) + sanitise; the UI shows "File N of M" for opaque ids. SAF folder trees aren't available, so Android "bulk" upload uses the picker's **multi-select**; desktop keeps folder walking.
- New plugins: `tauri-plugin-notification` + `tauri-plugin-fs` (Cargo + npm + registered in `lib.rs`); `notification:default` added to capabilities. `sha2` added to **both** server + client. `POST_NOTIFICATIONS` in the Android manifest. **Mobile nav:** Upload/Account/Create-account were sidebar-only (hidden on phones) — a [`MobileTopBar`](./src/components/MobileTopBar.tsx) overflow menu + a Home Upload tile expose them on Android. Tests: client 4 (sniff, filename, name-hint, chunk math), server 2 (id sanitisation, hex). **Deferred / notes:** sidecar `cover.jpg` is no longer sent with single-file uploads (the chunked protocol uploads only the file — embedded tag art + the server's own cover fetch still apply); the client buffers the whole file in native `Vec<u8>` to hash + slice chunks (fine for albums; true streaming range-reads via `app.fs().open()`+seek for multi-GB ISOs is a future optimisation); a true Android foreground service for uploads surviving process death is still deferred.

  3. **Prune** drops downloaded tracks whose local file vanished from disk.

**Phase 4 — Playback: complete.**

- New native module [`src-tauri/src/player/`](./src-tauri/src/player): a custom **`media://` URI-scheme protocol** (`stream.rs`) the webview's `<audio>` element loads directly, plus a `resolver.rs` URL helper.
  - Per request the protocol resolves the track id: **local cache hit** → serves `local_file_path` from disk with full RFC 7233 byte-range semantics (200 / 206 / 416, `Accept-Ranges`/`Content-Range`/`Content-Length`, MIME by extension); **cache miss** → proxies `GET /tracks/{id}/stream` from the server, injecting the active `Authorization` header and forwarding the incoming `Range` header, relaying the server's status + stream headers; **offline + not cached** → 502 so the element surfaces an error.
  - This keeps the session token out of the webview (the `<audio>` element can't set auth headers) and means the frontend never branches on online/offline — source resolution (prefer-local-else-stream) lives entirely in Rust. A process-wide `OnceLock<reqwest::Client>` backs the proxy so range requests don't rebuild TLS state.
  - Registered in `lib.rs::run()` via `register_asynchronous_uri_scheme_protocol("media", …)`; the handler is generic over the runtime and dispatches each request onto the async runtime.
- One Tauri command [`commands/player_commands.rs`](./src-tauri/src/commands/player_commands.rs): `player_media_url(track_id)` returns the platform-correct URL string (`media://localhost/<id>` on macOS/Linux/iOS, `http://media.localhost/<id>` on Windows/Android), defensively percent-encoded.
- Frontend: a Zustand [`src/player/store.ts`](./src/player/store.ts) owning the queue, current index, play state, position/duration, volume, **shuffle** (Fisher–Yates with the chosen track pinned first) and **repeat** (`off`/`all`/`one`); it binds to a single persistent `<audio>` element and drives `src`/play/pause/seek.
